## 1. RALPLAN-DR summary

### Principles (load-bearing)
1. **Zero extra tokens + zero OAuth are inviolable.** No always-on model calls; every model-touching feature is opt-in behind a flag and accounted for in token-accounting.
2. **One asset-naming scheme, one source of truth.** Installers, CI, and packaging must agree byte-for-byte on artifact names or the install path silently 404s (the existing `install.sh` TODO is exactly this drift). This includes killing *dead* alternate schemes (the second homebrew formula, the dead bun CI legs in *all three* jobs) — drift hides in artifacts the plan forgets to delete.
3. **Surgical diffs, existing style.** Reuse `apohara-indexer` (sqlite-vec + `feature_hash_embed`) and the `~/.apohara/` path convention rather than inventing infra. The `orchestration.db` stays new-per-run. *Caveat (resolved this session):* `cargo clippy --workspace --all-targets -- -D warnings` is **confirmed GREEN today** (Critic + Architect both ran it, exit 0), so the T-A0 clippy merge-gate does NOT trigger cross-crate warning-fixes — Principle 2 (CI invariant) and Principle 3 (surgical) are reconciled, not in tension.
4. **Pablo gates the release.** No `git push`, no tag cut, no `cargo publish`/AUR/Homebrew/Scoop push runs autonomously (standing rc.x gate). The matrix is wired; firing it is a human action.
5. **Don't overclaim — honesty of claim is a hard rule.** README softens "Arch-only" only once cross-platform install is genuinely exercisable; no TTFT number lands without a measured value; episodic recall is described as **feature-similarity recall (no model)**, never "semantic"; a GATE is only called a *CI* gate if CI actually runs it. Note: README:22's "lowers Time-To-First-Token" tagline is **intentionally retained as aspirational marketing copy** (not a measured metric) — only the README:211 under-promise is a hard guard.

### Decision Drivers (top 3)
1. **Adoption is blocked at the front door, AND the door has no working lock.** `install.sh` downloads a bare `apohara-desktop` asset the corrected matrix doesn't produce — *and* the PR-gating `ci.yml` is a dead TS/bun workflow whose **three jobs** (`test`, `npx-tarball-e2e`, `cross-platform-smoke`) all fail at `bun install` step 1, so no `cargo` regression is caught on PR. Both are "distribution unblock": you cannot responsibly publish from a repo whose CI is red. Highest impact / lowest effort. Phase A first, **CI resurrection before the new crate lands.**
2. **Honest differentiator with design-compatible cost** — episodic memory reuses existing zero-token, zero-model, local infra. It's the only "new capability" that doesn't risk a hard-rule violation. Recall is feature-hash similarity, *not* semantic — claimed honestly per Principle 5.
3. **Risk asymmetry** — Phase C (model-judge / TTFT bench) carries the highest hard-rule-violation risk for the lowest adoption value, so it's last, flag-gated, never default-on, **and its two surfaces (verification crate, bench script) are confirmed-dead today** — so each Phase-C task either builds the missing wiring explicitly or is de-scoped, never left fake-green.

### Viable Options

**Decision 1 — Sequencing: distribution-first vs feature-first**
- **Option 1A (CHOSEN): Distribution-first (A → B → C), with CI-resurrection as the very first Phase-A task (T-A0).**
  - Pros: unblocks adoption immediately; A is independent of B/C; **a working `cargo --workspace` CI guards Phase B's new crate the moment it exists**; a tag cut can publish working installers before episodic memory lands; smallest blast radius per phase. Clippy gate confirmed green today, so T-A0 stays one task (no A0a/A0b split needed — the Architect's split was contingent on clippy possibly being red; it is not).
  - Cons: episodic memory (the "differentiator") ships later; one extra task before the install.sh fix.
- **Option 1B (rejected): Feature-first (B → A).**
  - *Invalidation:* the audit marks distribution 🔴 blocking and episodic 🟡; shipping a differentiator behind a broken installer *and* red CI means nobody can install it and every regression in the new crate flies blind. Inverts the priority the verdict assigned. Rejected.

**Decision 2 — Episode store: new crate vs extend `apohara-indexer`**
- **Option 2A (CHOSEN): New thin crate `apohara-episodic` depending on `apohara-indexer`'s storage primitives (the now-`pub` `ensure_vec_extension_registered`, `feature_hash_embed`, `EMBED_DIM=384`).**
  - Pros: keeps `apohara-indexer`'s narrow "code chunks" surface clean (its `IndexedChunk` is `{file_path,start_line,end_line,body}` — episodes are `{goal,providers,verdicts,outcome}`, a different schema); isolates the cross-run DB path from the code index; ledger backend lives next to its data.
  - Cons: one more crate in the workspace (now 37); must re-solve vec0 registration (T-B3a/T-B3 own it explicitly — see Decision 6); **`feature_hash_embed` was tuned for long code `body` text, so feature-similarity recall over short one-sentence `goal` strings is keyword-ish, not semantic** — accepted and documented honestly (Principle 5), with a distance-ordering test (T-B4) to catch degeneracy.
- **Option 2B (rejected): Add an `episodes` table + functions directly inside `apohara-indexer`.**
  - *Invalidation:* conflates two unrelated stores (code index is per-repo, ephemeral-ish; episodes are cross-run, durable) in one crate, and would force `apohara-mcp`'s ledger backend to depend on the whole indexer. Cleaner to reuse the *embedding/vec0 primitives* (now `pub`) from a dedicated crate. Rejected — but note 2A still reuses, not reinvents, the sqlite-vec machinery.

**Decision 3 — Release workflow topology: unify vs keep two**
- **Option 3A (CHOSEN): Fold the desktop binary into the existing `release.yml` matrix and delete `desktop-release.yml`.**
  - Pros: `release.yml` already produces `apohara-<target>.tar.gz` + `.sha256` and packaging (`packaging/aur/PKGBUILD:23-24`, `packaging/scoop/*.json`) already references exactly that scheme. Adding `-p apohara-desktop-dioxus` to its existing `cargo build`/`cross build` steps and staging the extra binary is a 1-workflow change; one matrix, one naming scheme, one place to reason about.
  - Cons: `release.yml`'s Windows leg uses `zip`; the desktop binary rides along in the same archive (acceptable — TUI already does).
- **Option 3B (rejected): Rewrite `desktop-release.yml` as a separate Dioxus cargo matrix.**
  - *Invalidation:* two workflows producing release assets = two naming schemes to keep in sync = the exact drift class this remediation exists to kill. The Tauri/bun matrix is fully stale (no `package.json` in repo per README:217); there's nothing to preserve. Delete it. Rejected.

**Decision 4 (iteration 2) — Phase-C model-judge: wire dispatch inline vs prompt-build only**
- **Option 4A (CHOSEN): Prompt-build + flag plumbing only; model dispatch deferred as a documented follow-up.**
  - Pros: `apohara-verification` has zero spawn/dispatch/reqwest call sites today (`critic_prompt.rs:34` only renders a String; `run_all_gates:542` is pure regex over `GateInput`). Adding a model-dispatch dependency to the verification crate is real scope creep *and* hard-rule risk (it's the crate that must stay zero-token by default). Building the opt-in flag predicate + the prompt-assembly tier *now*, and leaving the actual provider round-trip as an explicit follow-up, keeps Phase C surgical and honest. **Safety-by-construction: no dispatch path exists in the gate crate, so a flag bug literally cannot spend tokens** (Critic verified: `run_all_gates` pure, zero spawn/reqwest in crate).
  - Cons: the judge tier doesn't produce a real verdict in v1 — it produces a *ready-to-dispatch prompt* gated behind `APOHARA_MODEL_JUDGE=1`. Documented as such; no fake APPROVE/REJECT plumbing.
- **Option 4B (rejected): Wire `build_critic_prompt` → a provider dispatch → parse `APPROVE|NEEDS_CHANGES|REJECT` back inline in `apohara-verification`.**
  - *Invalidation:* requires the verification crate to gain a CLI-spawn / dispatch dependency, which (a) couples a pure-regex gate crate to the provider layer, (b) risks the zero-token default if any code path forgets the flag, and (c) is several hundred LOC of scope beyond "optional/flagged." The plumbing does not exist; inventing it here violates Surgical Diffs. Rejected — deferred to a follow-up ADR.

**Decision 5 (iteration 2) — TTFT bench: re-target to Rust vs cut**
- **Option 5A (CHOSEN): Cut the real-TTFT mode from Phase C; keep README:211 under-promise verbatim.**
  - Pros: `scripts/bench/dispatch-latency.sh:26-30` launches `bun --hot src/server.ts` against the deleted `packages/desktop/` — it **cannot run at all today**. Re-targeting it to the Rust `apohara-desktop-dioxus` dispatch path is several hundred LOC of real work, clearly beyond "optional/flagged," and the adoption value is low (verdict's own assessment). Cutting it removes a fake-green verify and a dead script from the plan's surface.
  - Cons: no measured TTFT number this cycle (README stays honestly silent — which Principle 5 *requires* anyway).
- **Option 5B (rejected): Re-target the bench to the Rust dispatch path inside Phase C.**
  - *Invalidation:* turns an "optional/flagged" task into a multi-hundred-LOC bench rewrite driving the real coroutine path — scope creep, and it competes with the higher-value Phase A/B work. Deferred to a follow-up. Rejected.

**Decision 6 (NEW, iteration 3) — Episode-DB vec0 registration: assume indexer registered it vs own it explicitly**
- **Option 6A (CHOSEN): The episode crate OWNS its vec0 registration trigger by promoting `apohara-indexer::ensure_vec_extension_registered` to `pub` (T-B3a) and calling it from `open_episode_db` (T-B3) before any `Connection::open`.**
  - Pros: correct in a fresh process with no prior indexer call (the actual desktop-dispatch reality — the capture/recall path in T-B5/T-B6 has no guaranteed prior `open_db`); reuses the indexer's exact `OnceLock`-guarded `sqlite3_auto_extension` block (no duplicated `unsafe transmute`); single source of truth for the registration; documents the process-global side effect.
  - Cons: widens `apohara-indexer`'s public surface by one fn (acceptable — it's a registration primitive, not the chunks schema).
- **Option 6B (rejected): Duplicate the `unsafe sqlite3_auto_extension` transmute block in `apohara-episodic`.**
  - *Invalidation:* two copies of an `unsafe` FFI block to keep in sync, and a second process-global side-effect site — more drift surface for zero benefit since 6A reuses the existing primitive. Rejected.
- **Option 6C (rejected): Assume the indexer's `open_db` ran first in-process so vec0 is already registered.**
  - *Invalidation:* the desktop dispatch path (T-B5 capture, T-B6 recall) has NO guaranteed prior indexer call; in a fresh process `episodes_vec` vec0 creation fails at runtime. This is the single most likely Phase-B runtime failure (Architect's assessment). Rejected.

**Decision 7 (NEW, iteration 3) — Episode→LedgerEvent mapping: explicit lossy map vs reconsider the seam**
- **Option 7A (CHOSEN): Implement `LedgerBackend` with an explicit, documented lossy mapping; `replay_run`/`last_event` are degenerate-by-design; the integration test asserts non-empty returns from a self-seeded store.**
  - Pros: replaces the `EmptyLedger` stub (`api.rs:42`) with something that actually returns data for the methods that have episode meaning (`read_events`/`search_events`), and is honest about the two that don't (`replay_run`/`last_event`). Lowest-effort path that lands a real ledger backing without inventing a new MCP tool surface this cycle.
  - Cons: two of four trait methods are degenerate (return empty/best-effort); the episode model and the run-keyed ledger model remain a genuine impedance mismatch, papered over rather than resolved.
- **Option 7B (rejected this cycle, recorded as ADR follow-up): Give episodes their OWN MCP tool surface instead of masquerading as the run-keyed ledger.**
  - *Invalidation:* a new MCP tool surface is a larger design effort (new server, new schema, new bootstrap wiring) that exceeds this remediation's scope and competes with Phase A. The `EmptyLedger` stub is the documented placeholder the plan is meant to fill *now*; a cleaner seam is a deliberate follow-up, not a v1 blocker. Deferred — noted in ADR Follow-ups.

## 2. Pre-mortem (DELIBERATE) — 3 failure scenarios + mitigation

1. **Release matrix produces mismatched asset names → `install.sh` 404s on every platform, and red PR-CI never catches the regression — AND the CI fix itself can't go green because the verify scans the whole file.**
   - *Cause:* `install.sh` builds `apohara-desktop-${PLATFORM}` (os-arch slug like `linux-x64`) while `release.yml` emits `apohara-<rust-triple>.tar.gz`. Two naming universes. Worse, the *anti-drift guard* the plan relies on can't go green because `ci.yml` dies at `bun install` (step 1) before any guard runs — in **three** jobs (`:49`, `:185`, `:231`). And the iter-2 plan's own T-A0 verify (`! grep -q 'bun install --frozen-lockfile'` over the whole file) would itself FAIL because `:185`/`:231` survive a test-job-only edit.
   - *Mitigation:* **T-A0 enumerates and removes all three dead bun job-sets** (`test`, `npx-tarball-e2e`, `cross-platform-smoke`) and adds a live Rust CI job (`cargo build/test/clippy --workspace`). Its verify is **scoped to the `test` job via a YAML parse** (`'bun install --frozen-lockfile' not in str(jobs['test'])`) plus a whole-file assertion that *zero* bun-install legs remain, which is now true because all three are removed — so the verify is satisfiable. T-A1 rewrites `install.sh` to map `uname` → the **rust-triple** scheme `release.yml` already uses. T-A6 adds the triple-consistency guard **to the live job from T-A0** and asserts the `uname→triple` mapping function exists.

2. **Episode DB path collides across git worktrees, OR vec0 isn't registered in a fresh process → `episodes_vec` CREATE fails at runtime, OR concurrent UI runs contend writer-vs-writer → corrupt/raced cross-run store.**
   - *Cause:* (a) dispatch runs each provider in its own worktree (`dispatch_loop.rs:83`); a `current_dir()`-relative DB would fork per worktree. (b) vec0 registration is **process-global** (`sqlite3_auto_extension` behind a `OnceLock`, `storage.rs:52-77`) and PRIVATE; the desktop dispatch path (T-B5/T-B6) has no guaranteed prior indexer `open_db`, so opening `episodes.db` with no prior registration makes the vec0 virtual-table CREATE fail. (c) two `run_dispatch` invocations across UI sessions write the *same* `episodes.db` concurrently — WAL fixes reader-vs-writer, **not** writer-vs-writer (the `runSerialized` incident class in CLAUDE.md).
   - *Mitigation:* (a) episode DB path **anchored to `~/.apohara/episodes/episodes.db`** via a `dirs::home_dir()` helper mirroring `bootstrap.rs:48-69` — never `current_dir()`; test asserts cwd-independence. (b) **T-B3a promotes `ensure_vec_extension_registered` to `pub`; T-B3's `open_episode_db` calls it before any `Connection::open`**, with a **fresh-process test** (`serial_test`, matching the indexer's pattern) that opens `episodes.db` with NO prior indexer call and asserts `episodes_vec` creates; the process-global side effect is documented in the module doc. (c) open with **WAL + `busy_timeout = 5000ms`**; single-writer per run is guaranteed (capture runs once from the one `dispatch_loop` coroutine, after the provider loop ends at `:131`, before `set_status(Idle)` at `:137`); cross-session contention resolves via `busy_timeout` blocking, not corruption.

3. **`APOHARA_MODEL_JUDGE` flag accidentally defaults ON → silent token spend, hard-rule violation, automatic Critic REJECT.**
   - *Cause:* copying the `apohara-mcp` flag idiom where `is_enabled` defaults ON unless `=0` (`api.rs:26`) would invert the safe default for a model-calling feature.
   - *Mitigation:* the judge predicate uses **opt-IN** semantics (true ONLY for `Some("1")`; absent/`0`/anything-else = OFF), the inverse of the MCP flag. First test written asserts `judge_enabled(None)==false`, `judge_enabled(Some("0"))==false`, `judge_enabled(Some("1"))==true`. Crucially (Decision 4A), the judge tier in v1 only *assembles a prompt* — there is no inline model dispatch in `apohara-verification` (Critic verified: zero spawn/reqwest in the crate; `run_all_gates:542` pure), so even a flag bug cannot spend tokens from inside the gate crate. The 7 regex gates in `run_all_gates` are untouched and stay zero-token.

## 5. ADR

**Decision.** Remediate the audit's green-lit items in three ordered phases: (A) unblock distribution by **first resurrecting a live Rust CI** (`cargo build/test/clippy --workspace`, deleting the dead bun legs in *all three* `ci.yml` jobs — `test`/`npx-tarball-e2e`/`cross-platform-smoke`), then converging `install.sh`, the CI release matrix, and packaging on the single `apohara-<rust-triple>.{tar.gz,zip}` + `.sha256` scheme already used by `release.yml` — folding the desktop binary into that matrix (shipped-but-not-installed), deleting the stale Tauri `desktop-release.yml` *and the dead second homebrew formula `apohara.rb`*, plus dead-code cleanup; (B) add local episodic memory as a new `apohara-episodic` crate that reuses `apohara-indexer`'s sqlite-vec + `feature_hash_embed` primitives (feature-similarity recall, honestly *not* semantic), **owning its own process-global vec0 registration** via a newly-`pub` `ensure_vec_extension_registered`, hooks capture into `dispatch_loop::run_dispatch` at `:131-137`, wires retrieval, and replaces the `EmptyLedger` stub with an episode-backed `LedgerBackend` whose `run_id`-keyed methods are explicitly mapped or documented-degenerate; (C) add an opt-in model-judge **prompt-build tier** (flag plumbing only; actual dispatch deferred — the verification crate has no model-invocation path today) and **cut** the dead TTFT bench, both default-OFF / honesty-preserving.

**Drivers.** (1) Distribution is the 🔴 adoption blocker and is independent of features → first; *and* the PR-gating CI is itself dead in three jobs, so resurrecting it is part of "can't responsibly publish from red CI" → T-A0 leads. (2) Episodic memory is the only differentiator compatible with the zero-token/zero-OAuth/local hard rules → second. (3) Model-judge/TTFT carry the highest hard-rule risk for the lowest adoption value, and both Phase-C surfaces are confirmed-dead today → last, flagged, never default-on, each task either builds the missing wiring explicitly (T-C1/C2/C3) or is cut (T-C4/C5).

**Alternatives considered.** Feature-first sequencing (rejected — inverts the verdict's 🔴/🟡 priority and ships a differentiator behind a broken installer *and* red CI); extending `apohara-indexer` with an episodes table (rejected — conflates per-repo code index with cross-run episode store and over-couples the MCP ledger); keeping two release workflows / rewriting `desktop-release.yml` as a separate Dioxus matrix (rejected — two asset-naming schemes is the exact drift class being fixed; the Tauri workflow is fully stale); MCP-style default-ON flag for the judge (rejected — would silently spend tokens, automatic Critic REJECT); wiring model dispatch inline in `apohara-verification` (Decision 4B, rejected — no dispatch path exists; adding one couples a pure-regex gate crate to the provider layer, risks the zero-token default, and is multi-hundred-LOC scope creep → deferred follow-up); re-targeting the dead TTFT bench to the Rust dispatch path inside Phase C (Decision 5B, rejected — multi-hundred-LOC rewrite, beyond "optional/flagged", low adoption value → deferred follow-up); **duplicating the `unsafe sqlite3_auto_extension` block in `apohara-episodic` (Decision 6B, rejected — two `unsafe` FFI copies to sync; reuse the indexer primitive instead) and assuming the indexer pre-registered vec0 (Decision 6C, rejected — the desktop dispatch path has no guaranteed prior `open_db`, so a fresh process fails the vec0 CREATE)**; **giving episodes their own MCP tool surface instead of backing `LedgerBackend` (Decision 7B, rejected this cycle — larger design effort beyond this remediation; the `EmptyLedger` stub is the documented placeholder to fill now → deferred follow-up)**; bolting the anti-drift guard onto the dead bun `ci.yml` (rejected — it never reaches green; T-A6 now attaches to the live T-A0 job); installing the desktop GUI binary everywhere via packaging (rejected this cycle — runtime display requirements make silent install onto headless machines undesirable; shipped-but-not-installed, with everywhere-install a deliberate follow-up).

**Why chosen.** Each phase is independently shippable and verifiable, and—**post-T-A0—genuinely CI-guarded** (clippy `-D warnings` confirmed green today, so the gate is surgical) before the new `apohara-episodic` crate lands; the chosen distribution topology reuses the scheme packaging already expects (`packaging/aur/PKGBUILD:23-24`, `packaging/scoop/*.json`), minimizing diff; episodic memory reuses proven local infra (`apohara-indexer` storage, `~/.apohara/` path convention from `bootstrap.rs:48-69`), **owns its vec0 registration so it cannot fail in a fresh dispatch process**, and lands the real ledger the `api.rs:42` stub was always a placeholder for (with the run-keyed impedance mismatch handled by an explicit, documented mapping rather than fake-empty returns); the opt-in inverse-flag default + prompt-build-only scoping make the hard-rule-critical Phase C safe by construction (a flag bug cannot spend tokens because no dispatch path exists in the gate crate).

**Consequences.** Workspace grows 36→37 crates (`apohara-episodic`). `apohara-indexer` gains one `pub` registration fn (visibility-only; no behavior change). `ci.yml` gains a live Rust `--workspace` job and sheds its dead bun legs in all three jobs; every GATE becomes a real PR invariant. One unified release workflow (`release.yml`) owns all assets including the desktop binary (shipped-but-not-installed). The dead second homebrew formula is removed. A new durable cross-run DB at `~/.apohara/episodes/episodes.db` (WAL + `busy_timeout=5000ms`, single-writer per run, serialized across sessions, process-global vec0 registration documented) — distinct from and not altering the new-per-run `orchestration.db`. Feature-hash recall over short goals is keyword-ish, documented as such (no "semantic" claim). The `EpisodicLedger`'s `replay_run`/`last_event` are degenerate-by-design (episodes lack run_id). The Phase-C judge produces a prompt, not a verdict, in v1. README:22's TTFT tagline is intentionally retained as marketing copy; only :211 is guarded. The tag cut (T-A9) and any AUR/Homebrew/Scoop/crates.io pushes remain Pablo-gated manual actions; this plan does not automate them.

**Follow-ups (deferred / out of scope).** (1) **Model-judge dispatch**: wire `build_critic_prompt`'s output to a provider round-trip and parse `APPROVE|NEEDS_CHANGES|REJECT` back into a `GateBlock` (needs a dispatch path the verification crate currently lacks; the T-C3 `Counter::record_absolute` hook is pre-placed for it). (2) **Re-target the TTFT bench** to the Rust `apohara-desktop-dioxus` dispatch path (replacing the dead `bun --hot src/server.ts`), then measure and update README:211/:22. (3) **Dedicated MCP tool surface for episodes** (Decision 7B) — replace the degenerate `replay_run`/`last_event` semantics with a goal/timestamp-native query surface instead of masquerading as the run-keyed ledger. (4) **Install the desktop GUI binary** via packaging once headless-safe install gating is designed (currently shipped-but-not-installed). (5) SHA256 sidecar resolution for AUR `PKGBUILD` (currently `SKIP`) at real tag time. (6) Potential migration of the per-run orchestration ledger into the episode store (not now — orchestration.db stays new-per-run by design). Explicitly rejected and NOT planned: SPLADE/ColBERT, EDF scheduling, always-on LLM judge, web UI, plugin marketplace, 4th provider, cargo-zigbuild.
