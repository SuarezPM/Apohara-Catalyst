# G5.A.12 — enum_dispatch + spawn_blocking decisions

> Auto Mode decision recorded 2026-05-23 during Apohara Ultimate Sprint 5
> Wave 3 (G5.A.12). Audit had these two items flagged AMBIGUO; this doc
> closes them.

## Context

The vibe-kanban audit flagged two Rust idioms as "PARCIAL/AMBIGUO" without a
concrete adoption verdict:

1. **enum_dispatch** — should we replace `Box<dyn Trait>` boxes with
   `enum_dispatch` to get monomorphic dispatch and skip the vtable cost?
2. **spawn_blocking** — should we wrap any heavyweight sync work inside
   tokio handlers with `tokio::task::spawn_blocking`?

Both are valid patterns. The question is whether the current Apohara code
needs them today.


Only one `Box<dyn ...>` site in the entire workspace and it's the sandbox
`Profile` trait. Both `#[cfg]`-gated arms (`linux::LinuxProfile`,
`fallback::FallbackProfile`) are mutually exclusive at compile time — only
ONE concrete type exists in the final binary for a given target.

No tokio handler in the workspace currently does any CPU-bound work inside
the async path. The Nomic-BERT indexer is sync (mock mode in CI, real model
runs through a dedicated executor on the host).

## Decision

### enum_dispatch — NOT adopted for v1.0

**Why:** `enum_dispatch` is a win when a single binary needs to dispatch
over multiple concrete variants at runtime without paying the vtable
cost. Our `Profile` site has only ONE variant per platform after
preprocessing; the `Box<dyn>` indirection there is essentially free —
the call gets inlined by LLVM when the target is known.

Adding `enum_dispatch` would require a single enum carrying both
`LinuxProfile` and `FallbackProfile`, defeating the `#[cfg]` build-time
gating that keeps the non-Linux build from compiling Linux-specific
seccomp code at all.

**Action:** keep `Box<dyn Profile>` for now. Spec §0.16 still recommends
`enum_dispatch` as the default for **new** trait sites with multiple
runtime variants — this is the SHOULD, not a retroactive rewrite of
single-site code where `#[cfg]` already does the dispatch.

### spawn_blocking — NOT needed for v1.0

**Why:** No tokio executor inside the workspace is currently parking on
a sync syscall hot enough to need `spawn_blocking`. The hooks-server is
pure async I/O; the indexer is invoked from a dedicated process; the
audit/notifications/worktree crates do quick sync work that finishes in
<1ms.

**Action:** document the pattern as the preferred wrapper for any
future CPU-bound sync work that lands inside a tokio runtime (e.g. if
the indexer ever runs inline with the hooks-server in v1.x). No
implementation change needed in v1.0.

## How to apply if circumstances change

1. **New trait with 2+ runtime variants** in the same binary →
   declare with `#[enum_dispatch]`:

   ```rust
   #[enum_dispatch::enum_dispatch]
   pub trait Driver {
       fn name(&self) -> &str;
   }
   #[enum_dispatch::enum_dispatch(Driver)]
   pub enum AnyDriver {
       Claude(ClaudeDriver),
       Codex(CodexDriver),
   }
   ```

2. **CPU-bound sync work inside a tokio handler** → wrap with
   `spawn_blocking`:

   ```rust
   let result = tokio::task::spawn_blocking(move || {
       expensive_sync_work(input)
   }).await?;
   ```

   Match max blocking-thread budget against the deployment's CPU count
   (see `tokio::runtime::Builder::max_blocking_threads`).
